### 修复支付宝扫码支付 404 + API 500
**根因1（支付宝扫码 404）：** `.env` 中支付宝私钥使用了多行格式（每行一个 `
`），但 dotenv 无法解析多行值，导致只读取了第一行 `-----BEGIN PRIVATE KEY-----`（27个字符），后续所有内容被丢弃。`normalizePem` 函数只拿到了一个不完整的 PEM header，私钥签名时抛错 `1E08010C-DECODER routines::unsupported`。

**根因2（余额充值支付宝 mock URL）：** `balance.controller.js` 的 `createRechargeOrder` 直接自己生成 mock URL，没有调用 `alipayService.createPrecreateOrder()`，所以当 ALIPAY_MOCK=0 时仍然会返回模拟地址。

**根因3（所有 API 500）：** `billing_invoice_requests` 表被早期 migration 068 创建了一个完全不同的 schema（`invoice_type` enum personal/corporate、`tax_number` 列名、缺少 `requested_by`/`email`/`issued_at` 等 7 个列），而 Sequelize model 期望的是 094 migration 的 schema，导致 SELECT/INSERT 全部失败。

**修复内容：**
1. 支付宝私钥和公钥转为单行 `\n` 格式写入 `.env`（`normalizePem` 会还原为真实换行）
2. 从 zxai/platform/certs/alipay/ 复制现成密钥文件到 syqw backend/certs/alipay/
3. 重写 `balance.controller.js` createRechargeOrder()：微信支付调用 `wechatPayService.createNativeOrder()`，支付宝调用 `alipayService.createPrecreateOrder()`，真正走支付网关
4. `alipay.service.js` mock URL 改为 `alipays://platformapi/startapp?appId=...`（避免 `https://qr.alipay.com/MOCK_` 被浏览器 404）
5. 修复 `billing_invoice_requests` 表结构（ALTER TABLE）：添加 7 个缺失列 + 修改列名/ENUM/长度
6. 验证：所有端点 401（鉴权正常）而非 500/404 ✅

### ECS 部署教训（2026-06-02 下午）
- **ECS 项目路径：** `/var/www/wework-saas/`（后端源码 + 前端源代码）
- **Nginx 静态文件路径：** `/var/www/wework/`（前端 dist 输出）
- **PM2 进程：** syqw-api（cluster 2实例），端口 3010
- **部署流程：**
  1. 本地 `npm run build` 构建前端 dist
  2. 打包 `tar czf` 包含 backend/src/ + frontend/dist/ + database/
  3. 通过阿里云 Workbench「文件上传」传到 ECS（路径在 Workbench 的 webroot）
  4. ECS 解压到 `/var/www/wework-saas/`
  5. `cp -rf /var/www/wework-saas/frontend/dist/* /var/www/wework/`
  6. `pm2 restart syqw-api --update-env && systemctl reload nginx`
- **关键认知：** Mac → ECS SSH 直连不通（阿里云安全组拦截），必须通过 Workbench 终端操作
- **浏览器缓存：** curl 只能验证静态 HTML（title 不变），React 动态内容需浏览器无痕模式查看
